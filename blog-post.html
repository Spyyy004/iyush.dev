Building AuthWatch: reading the GoHighLevel API

Every GoHighLevel agency runs email for dozens of sub-accounts, and every one of those sub-accounts can quietly break its own SPF, DKIM, or DMARC records. When that happens, deliverability craters and nobody notices until a client complains. AuthWatch exists to notice first.

The plan was simple: read each sub-account's sending domain from the GHL API, resolve its DNS, and diff the records against what they should be. The API had other ideas.

What the API exposes

The GHL v2 API is generous with contacts, conversations, and calendars. It is far quieter about infrastructure. Listing locations (sub-accounts) is straightforward:

// list every sub-account under the agency token
const res = await fetch("https://services.leadconnectorhq.com/locations/", {
  headers: {
    Authorization: `Bearer ${agencyToken}`,
    Version: "2021-07-28",
  },
});
const { locations } = await res.json();

But the field I actually wanted — the sub-account's LC Email sending domain — isn't there. You can read a location's business profile, its settings, even its custom values, and the sending domain simply is not part of the payload.

The API tells you who the client is. It does not tell you how their mail leaves the building.

Designing around the gap

Rather than block on an endpoint that doesn't exist, AuthWatch treats the domain as an input, not a lookup. During onboarding, the agency confirms each sending domain once. From there the monitoring loop is pure DNS and needs nothing further from GHL:

async function checkDomain(domain) {
  const [spf, dkim, dmarc] = await Promise.all([
    resolveTxt(domain),
    resolveTxt(`ghl._domainkey.${domain}`),
    resolveTxt(`_dmarc.${domain}`),
  ]);
  return diffAgainstExpected({ spf, dkim, dmarc });
}

A few things fell out of this that I liked:

  • The check is provider-agnostic — it works whether the domain sends through GHL, Mailgun, or something else entirely.
  • It's cheap. DNS resolution is fast and rate-limit-free compared to hammering a partner API.
  • When a record breaks, I already know the expected value, so the alert can say add v=spf1 include:... instead of just "something's wrong."

The exact-fix alert

The difference between a useful monitor and an annoying one is whether it tells you what to do. AuthWatch never says "DMARC failing." It says: here is the record we found, here is the record we expected, here is the one line to paste into your DNS.

The lesson generalizes past GHL: when an API won't give you a field, ask whether you actually need it from the API — or whether the user can hand it to you once and let you own the rest of the loop. Designing around the gap shipped faster than waiting for the gap to close.
